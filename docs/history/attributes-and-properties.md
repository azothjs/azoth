# Attributes and Properties

> **Static attributes compile to HTML. Dynamic interpolation assigns DOM
> properties.** Two different mechanisms living under one syntactic surface.
>
> I read `<input name="title" value={x} />` as if it were uniform — one
> JSX element, one set of attributes. It's not. The static `name="title"`
> is baked into a `<template>` and parsed by the browser. The dynamic
> `value={x}` is a runtime line that does `el.value = x`. Same line of JSX,
> two different machines.
>
> Once I held that split, the `class` / `className` thing stopped feeling
> like a quirk and started feeling like the only thing it could be. Static
> goes to the HTML parser, which knows the word `class`. Dynamic goes to a
> DOM property assignment, which doesn't — the property is `className`.
> The syntactic surface is the same; the underlying mechanism is not.

## Static attributes: compiled into HTML

Any attribute with a literal value (no `{…}` interpolation) is extracted
into the static HTML template at compile time:

```jsx
<input name="title" required maxlength="80" />
```

becomes, in the emitted template:

```html
<input name="title" required maxlength="80" />
```

The browser's HTML parser handles it. All standard HTML applies — void
elements, boolean attributes, data-attributes, ARIA attributes, anything
the parser accepts. The names you write are the names the parser sees.
Use HTML attribute names here: `class`, `for`, `tabindex`, `readonly`.

## Dynamic interpolation: DOM property assignment

Any attribute with a JSX expression value becomes a runtime property
assignment generated by Thoth:

```jsx
<input value={title} />
```

The runtime, conceptually, does:

```js
el.value = title;
```

Not `el.setAttribute('value', title)`. This matters when the attribute
name and the DOM property name differ — which they do in several common
cases.

## The `class` / `className` foot-gun

This is the canonical instance. Static `class` works. Dynamic `class`
silently drops the value, because `el["class"] = …` is not a thing on a
DOM element. Use `className` for dynamic:

```jsx
<div class="card">…</div>              // static — works
<div class={extra}>…</div>             // dynamic — DROPS silently
<div className={extra}>…</div>         // dynamic — works
```

Mixed usage on the same element is fine — and often the right shape.
Static `class` for the base, dynamic `className` for the variant:

```jsx
<div class="card" className={`card ${variant}`}>…</div>
```

The static attribute ships in the HTML template (no flash of unstyled
content while JS loads). The dynamic property assignment runs at clone
time and overwrites with the full computed value.

## Common attribute → property mappings

When binding dynamically, use the DOM property name:

| HTML attribute | DOM property |
| -------------- | ------------ |
| `class`        | `className`  |
| `for`          | `htmlFor`    |
| `readonly`     | `readOnly`   |
| `tabindex`     | `tabIndex`   |

For attributes whose names already match the property (`value`, `name`,
`id`, `href`, `disabled`, `checked`, most others), there's nothing to
remember — write the same name in both cases.

## SVG presentation attributes can't be dynamically bound

SVG presentation attributes — `stroke`, `stroke-width`, `stroke-dasharray`,
`stroke-dashoffset`, `fill`, `transform`, and others — are exposed on SVG
elements as **read-only `SVGAnimated*` objects** (e.g. `SVGAnimatedLength`,
`SVGAnimatedString`). Assigning to them silently does nothing:

```jsx
// Value drops silently — the rendered SVG has no stroke-dasharray
<circle stroke-dasharray={dashArray} />
```

Static SVG attributes are fine — literal values go into the HTML template
and the parser handles them. It's the dynamic binding that fails.

**Workaround:** bind via `style` — `style.cssText` is writable, and CSS
understands SVG presentation properties:

```jsx
<circle style={`stroke-dasharray: ${dashArray}; stroke: ${color}`} />
```

For geometry attributes like `polyline.points` and `path.d` (read-only
`SVGPointList` / `SVGPathData` objects), build the SVG statically and use
`setAttribute` directly. See
[known-limitations](known-limitations.md) for the full treatment.

## Event handlers

Event handlers are just DOM properties — `onclick`, `oninput`,
`onkeydown`, etc. — so they work through the same dynamic-interpolation
mechanism:

```jsx
<button onclick={() => count++}>Click</button>
<input onkeydown={e => { if (e.key === 'Enter') submit(); }} />
```

The handler is assigned to the property directly. The event object the
handler receives is the **native** `Event` — not a synthetic wrapper.
`e.currentTarget`, `e.preventDefault()`, `e.stopPropagation()`, all the
standard DOM event API, available unchanged.

No synthetic event layer. No event pooling. No `onClick`-camelCase
convention — use the lowercase property names the DOM uses
(`onclick`, not `onClick`).

## Boolean shorthand on components

When passing a prop to a component, omitting the value defaults to `true`:

```jsx
<MyComponent flag />
// equivalent to
<MyComponent flag={true} />
```

This matches HTML's boolean-attribute convention and is convenient for
feature flags and toggles.

## Why this split exists

Static attributes go through the browser's HTML parser, which uses
attribute names. Dynamic bindings are JavaScript assignments to DOM
elements, which use property names. Azoth doesn't paper over the
distinction — it exposes it directly, because the cost of pretending the
two layers are the same shows up later as confusion (or, in the React
case, as a transpile-time keyword check that obscures what's actually
happening).

The contrast with React: React requires `className` and `htmlFor` because
`class` and `for` are reserved keywords in the JavaScript the JSX
transpiles into. Azoth's reason is different — static attributes never
touch the JS runtime as property names, so there's no keyword problem to
guard against. The reason `className` is required for *dynamic* bindings
is purely the DOM-property-name fact.

## Future work

A translation layer could let developers write `class={…}` for dynamic
bindings and have the compiler translate to `className=`. Currently you
have to pick the right name yourself. Same with `for` / `htmlFor` and
the rest of the small mapping table above.

This is on the roadmap. Until then, the table is the table.
